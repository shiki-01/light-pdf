# light-pdf 要件定義書

PDF を軽量化するクライアントサイド完結型 Web サービスの要件定義。

- 作成日: 2026-07-17 / 最終更新: 2026-07-18
- ステータス: 確定（実装前検証項目はセクション 8 に記載）
- 公開先: https://light-pdf.pages.dev （Cloudflare Pages）

---

## 1. コンセプト

- ブラウザ内で PDF・画像を結合・軽量化・編集し、1 つの PDF として出力する。
- ファイルは一切サーバーに送信しない（プライバシーが最大の差別化要素）。
- PC・スマートフォンの両方で全機能が利用できる。
- 言語は日本語のみ。

## 2. UI 設計方針（二層構造）

ライトユーザーとこだわるユーザーの両立を、基本画面と詳細設定の二層で実現する。

### 基本画面（デフォルト動線）

何も考えずに「言われたとおりにやれば」最低限の成果物ができることを最優先とする。

1. ファイルをドロップ（またはタップで選択）
2. プリセットを選ぶ（デフォルト: バランス）
3. ダウンロード

基本画面に露出する設定は**プリセット選択のみ**とする。

| プリセット | 内容 |
|---|---|
| 最軽量 | 長辺 1280px / JPEG 品質 60 / メタデータ削除オン |
| バランス（デフォルト） | 長辺 1920px / JPEG 品質 80 / メタデータ削除オン |
| 高品質 | 長辺 2560px / JPEG 品質 90 / メタデータ削除オン |

### 詳細設定（折りたたみ）

「詳細設定」を開くと、セクション 3.2 の全設定項目を個別に変更できる。プリセットは詳細設定の値をまとめて書き換えるショートカットとして機能し、詳細設定を個別に触るとプリセット表示は「カスタム」になる。

原則: **すべての挙動は詳細設定まで潜れば変更可能にする。基本画面には出さない。**

## 3. 機能要件

### 3.1 入力

| 項目 | 内容 |
|---|---|
| 入力方法 | ドラッグ&ドロップ、ファイル選択ダイアログ、クリップボード貼り付け（Ctrl+V） |
| PDF | 複数ファイル可。各ファイルをページ単位に分解して読み込む |
| 画像 | JPEG / PNG / WebP / HEIC / GIF（1 フレーム目）。各画像を 1 ページとして扱う |
| 画像の向き | EXIF Orientation を反映して取り込む |
| 追加入力 | 読み込み後も追加ドロップでページを末尾に追加できる（並べ替えも可） |
| 暗号化 PDF | パスワード入力ダイアログを表示。解除できない場合はそのファイルのみスキップし、他は継続 |
| 破損 PDF | 読み込み失敗はファイル単位でエラー表示し、処理全体は止めない |

- スマートフォンではドラッグ&ドロップが使えないため、タップでのファイル選択（`capture` なしの `<input type="file" multiple>`）を主動線とする。
- HEIC はブラウザネイティブでデコードできないため、WASM デコーダー（libheif 系）を遅延ロードで同梱する。HEIC が投入されるまでダウンロードしない。

### 3.2 軽量化処理

軽量化は 2 つのモードを提供する。ページ内容の種類によって最適解が異なるため、モード選択で「画像でないページをどう扱うか」を解決する。

#### モード A: 構造保持モード（デフォルト）

テキスト・ベクター要素はそのまま維持し、埋め込み画像のみを処理する。テキストの選択・検索・コピーが可能なまま軽量化できる。

- PDF 内の埋め込み画像を抽出し、長辺が指定値（デフォルト 1920px）を超えるものを縮小
- JPEG（品質指定可、デフォルト 80）へ再エンコード。透過（SMask）を持つ画像は透過を維持できる形式（Flate 圧縮）で再格納
- 再エンコード後のサイズが元より大きくなる場合は元データを維持（サイズ増加を防ぐ）
- テキスト・ベクター要素は無変換

#### モード B: ラスタライズモード

ページ全体を Canvas にレンダリングし、1 枚の画像に変換して再構成する。「Figma に画像として置いてフレームでエクスポートすると軽くなる」の再現に相当する。

- 各ページを長辺が指定値になる解像度でレンダリングし、1 枚の画像のページに置換
- 複雑なベクターデータ・重いフォントを含む PDF で効果が大きい
- テキストの選択・検索は失われる（UI 上で明示的に警告する）
- ページごとにモードを切り替え可能とする（全体一括設定 + ページ単位の上書き）

**ラスタライズ時の画像フォーマットは自動選択とする（決定事項）。**

- レンダリング結果を解析し、写真的なページ → JPEG、文字・線画中心のページ（色数が少ない / エッジが鋭い）→ Flate（PNG 相当）を自動で選ぶ
- 判定方法の例: 縮小サンプルの色数・両フォーマットで試し圧縮したサイズ比較。実装時に精度とコストのバランスで決定する
- 詳細設定で「自動 / JPEG 固定 / PNG 固定」を手動指定可能

#### 共通: 不要データ削除

モードに関わらず以下を削除・最適化する。

- ドキュメント情報辞書（Info: 作成者、作成アプリ、日時等）
- XMP メタデータ
- 埋め込みサムネイル
- 未使用オブジェクト（どこからも参照されないオブジェクトの除去）
- 重複オブジェクトの統合（同一画像・同一フォントの参照共有）
- 埋め込み JavaScript、フォームアクション
- 添付ファイル
- 無圧縮ストリームの Flate 圧縮
- オブジェクトストリーム・クロスリファレンスストリームによる保存
- フォントのサブセット化は初期リリースでは対象外（技術難度が高い。未使用フォントの削除のみ行う）

**出力 PDF バージョンは 1.7 とし、旧リーダー互換は考慮しない（決定事項）。** オブジェクトストリーム・クロスリファレンスストリームは常時有効とする。

注意: タグ構造（アクセシビリティ情報）や注釈の削除はデフォルトでは行わず、オプションとする。

#### 設定項目（詳細設定に配置）

| 設定 | デフォルト | 範囲 |
|---|---|---|
| プリセット | バランス | 最軽量 / バランス / 高品質 / カスタム |
| 長辺最大サイズ | 1920px | 480〜4096、または「縮小しない」 |
| JPEG 品質 | 80 | 10〜100 |
| モード | 構造保持 | 構造保持 / ラスタライズ |
| ラスタライズ時フォーマット | 自動 | 自動 / JPEG / PNG |
| グレースケール変換 | オフ | オン / オフ |
| メタデータ削除 | オン | オン / オフ |
| 注釈・タグ構造の削除 | オフ | オン / オフ |
| 出力ファイル名 | 最初のファイル名 + `_light.pdf` | 任意文字列 |

設定値は localStorage に保存し、次回訪問時に復元する（詳細設定から「デフォルトに戻す」可能）。

### 3.3 ページ編集

| 機能 | PC | スマートフォン |
|---|---|---|
| 並べ替え | サムネイルのドラッグ&ドロップ | 長押し後ドラッグ、または選択して「前へ / 後ろへ」ボタン |
| 回転 | ホバー時に表示されるボタン（90° 単位） | サムネイル選択時に表示される常設ボタン |
| 削除 | 同上 | 同上 |
| 複数選択 | Ctrl / Shift + クリック | 選択モード切替後にタップ |
| プレビュー | サムネイルクリックで拡大表示 | タップで拡大表示 |

ホバー依存の UI は禁止とし、すべての操作に「ホバーなしで到達できる導線」を必ず用意する（タッチデバイス判定ではなく `@media (hover: hover)` で出し分け）。

### 3.4 出力

- 全ページを 1 つの PDF（バージョン 1.7）に結合してダウンロード
- 出力前に「元の合計サイズ → 出力サイズ（削減率）」を表示
- スマートフォンでは Web Share API による共有も提供（対応ブラウザのみ）
- オプション: ページごとに個別 PDF として ZIP 出力（優先度: 低）

### 3.5 処理中の UI

- 処理は Web Worker で実行し、メインスレッドをブロックしない
- ページ単位の進捗表示（n / N ページ）とキャンセルボタン
- 処理完了前でもサムネイル一覧の閲覧・編集操作は可能とする

## 4. 非機能要件

### 4.1 プライバシー・セキュリティ

- ファイルデータの外部送信を行わない。fetch 先は自ホストの静的アセットのみ
- アナリティクスを入れる場合もファイル内容・ファイル名は送信しない
- 処理完了後、ユーザーがページを離れたらメモリ上のデータは破棄される（永続化しない）

### 4.2 パフォーマンス・リソース

- 目安: 50MB / 100 ページ程度の PDF を PC で 30 秒以内、スマートフォンで 90 秒以内に処理
- ピークメモリを抑えるため、全ページ同時展開ではなくページ単位の逐次処理とする
- iOS Safari は Canvas サイズ・メモリ制限が厳しいため、レンダリング解像度に上限を設ける
- 入力サイズの推奨上限（例: 合計 200MB）を超える場合は警告を表示（処理自体はブロックしない）
- WASM デコーダー（HEIC 等）は必要になるまで遅延ロードし、初期表示を軽く保つ

### 4.3 対応環境

- Chrome / Edge / Firefox / Safari の各最新 2 バージョン
- iOS Safari 16+、Android Chrome
- 画面幅 360px〜 でレイアウト崩れなし（レスポンシブ）
- オフライン動作: PWA として実装し、2 回目以降はネットワークなしで利用可能（Service Worker でアセットをキャッシュ）

### 4.4 アクセシビリティ

- 主要操作はキーボードのみで完結可能（並べ替え含む: 選択 + 矢印キー移動）
- タッチターゲットは 44px 以上
- ダークモード対応（`prefers-color-scheme` 追従）

## 5. 技術スタック（決定）

| 層 | 採用 | 用途 |
|---|---|---|
| フレームワーク | SvelteKit + TypeScript（adapter-static、全ページプリレンダリング） | UI。サーバー処理がないため完全静的サイトとして出力 |
| PDF レンダリング | pdf.js | サムネイル生成、ラスタライズモード、暗号化 PDF の解除読み込み |
| PDF 構造操作 | pdf-lib | 分解・結合・回転・画像差し替え・メタデータ削除・保存 |
| 画像処理 | OffscreenCanvas + `canvas.toBlob()` | 縮小・JPEG / PNG 再エンコード |
| HEIC デコード | libheif 系 WASM（遅延ロード） | HEIC 入力対応 |
| 並列化 | Web Worker（Comlink 等） | 処理のオフロード |
| ホスティング | Cloudflare Pages（kishintorii@gmail.com アカウント、light-pdf.pages.dev） | 静的配信 |

### ライブラリ選定の根拠と方針

- pdf.js + pdf-lib はいずれも寛容なライセンス（Apache-2.0 / MIT）であり、WASM 同梱による初期ロード増もない。
- 未使用オブジェクト削除は、pdf-lib で「新規ドキュメントに必要ページのみコピーして保存する」方式で実現する（参照されるオブジェクトのみがコピーされるため、実質的なガベージコレクションになる）。
- 重複統合は画像・フォントストリームのハッシュ比較で自前実装する。
- オブジェクトストリーム化は pdf-lib の `useObjectStreams: true` を使用する。
- 構造保持モードの画像差し替えは pdf-lib の低レベル API（`PDFRawStream` 等）で行う。
- **フォールバック**: 実装前検証（セクション 8）で pdf-lib の能力不足が判明した場合は mupdf.js (WASM) に切り替える。その場合 AGPL ライセンスとなるため、リポジトリの公開（オープンソース化）を前提とする。

## 6. デプロイ・運用

- Cloudflare Pages に GitHub リポジトリ連携でデプロイ（main ブランチ → 本番、PR → プレビュー）
- ビルド: `npm run build`（SvelteKit adapter-static の静的出力）
- カスタムドメインは当面使用せず light-pdf.pages.dev で運用

## 7. 追加機能（優先度付きバックログ）

| 優先度 | 機能 | 理由 |
|---|---|---|
| 高 | 軽量化前後のプレビュー比較（ページ拡大時に元 / 後を切替） | 品質劣化をユーザーが確認してから保存できる |
| 中 | ページ範囲の抽出（特定ページだけ出力） | 分解 UI がすでにあるため実装コストが低い |
| 中 | 白紙ページの検出・削除候補の提示 | スキャン PDF で需要が大きい |
| 低 | ページ番号の追記 | 結合ユースケースと相性が良い |
| 低 | 目標サイズ指定（「10MB 以下にしたい」→ 品質を自動探索） | メール添付上限ユースケース。実装は反復処理が必要 |

（プリセット・設定保存・HEIC 対応は本編要件に昇格済み）

## 8. 実装前検証項目（スパイク）

実装着手前に小さな検証コードで以下を確認する。不成立の場合はセクション 5 のフォールバックを適用する。

1. pdf-lib で既存 PDF の埋め込み画像（DCTDecode / FlateDecode、SMask 付き含む)を差し替えられるか
2. 「新規ドキュメントへのページコピー + `useObjectStreams: true`」でどの程度サイズが削減されるか（実 PDF 数種で計測）
3. pdf.js のレンダリング結果を Worker（OffscreenCanvas）内で完結できるか（iOS Safari 含む）
4. ラスタライズ時の JPEG / PNG 自動判定の精度とコスト（試し圧縮方式で十分か）

## 9. 決定事項の記録

| 論点 | 決定 |
|---|---|
| PDF 操作ライブラリ | pdf.js + pdf-lib。不足時は mupdf.js へフォールバック（AGPL 前提） |
| ラスタライズ時のフォーマット | 自動選択 + 詳細設定で手動指定可 |
| 旧リーダー互換 | 捨てる。PDF 1.7 / オブジェクトストリーム常時有効 |
| 対応言語 | 日本語のみ |
| サービス名・公開先 | light-pdf / light-pdf.pages.dev（Cloudflare Pages） |
| UI 方針 | 基本画面はプリセット選択のみ。全設定は詳細設定に集約 |
| フレームワーク | SvelteKit（adapter-static） |
